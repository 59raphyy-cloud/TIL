# 관통프로젝트 6회차
## 💬 데이터 크롤링 프로젝트
### 1. 프로젝트 개요
- 특정 기업의 댓글 데이터를 수집하여 정적 전처리(규칙 기반)와 LLM 기반 부적절 필터링을 수행하고, 정제된 데이터를 바탕으로 AI가 유사한 의미의 문장을 생성(증강)하여 풍부한 데이터셋을 구축하는 Django 웹 애플리케이션

### 2. 요구사항별 구현 내용
**1. [F101] 회사명 입력 및 검증 기능 구현**
- 사용자로부터 분석 대상인 회사명을 입력받는 메인 페이지 구현
- 입력값이 없는 경우 데이터 수집 절차를 진행하지 않도록 백엔드 유효성 검사 로직 적용

**2. [F102] 각 단계의 결과 저장**
- 수집 및 전처리 단계마다 발생하는 중간 결과물을 세션(Session) 또는 임시 리스트에 저장하여 데이터의 손실 없이 다음 단계(증강)로 전달되도록 설계함
- 각 단계별 데이터 상태(Raw -> Cleaned -> Augmented)를 구분하여 관리함으로써 추후 트러블슈팅 및 검증이 용이하도록 구현함

**3. [F103] 회사 검색 및 대상 회사 선택**
- 사용자가 입력한 키워드를 바탕으로 네이버 뉴스 또는 증권 API를 호출하여 정확한 업체명을 검색하고, 그중 최상단 결과를 분석 대상 기업으로 자동 매칭함
- 검색 결과가 여러 개인 경우, 가장 관련도가 높은(첫 번째) 기업의 고유 코드나 식별자를 추출하여 수집 단계로 전달함

**4. [F104] 커뮤니티 댓글 데이터 수집**
- 선택된 기업을 키워드로 하여 네이버 종목 토론실 등 주요 커뮤니티의 실시간 댓글 데이터를 수집함
- requests와 BeautifulSoup을 활용하여 HTML 구조를 파싱하고, 정적/동적 페이지 특성에 맞춰 안정적으로 텍스트 데이터를 추출하도록 구현함

**5. [F105] 데이터 전처리**
1) 정적 전처리
- 댓글 데이터 중 결측치, 너무 짧거나 긴 문장(5자 미만, 100자 초과)을 1차적으로 필터링
- 정규표현식을 활용하여 특수문자를 제거하고, 'ㅋㅋㅋㅋ, ㅎㅎㅎㅎ' 등 의미 없는 자모음 반복 패턴을 감지하여 제거
2) LLM 기반 부적절 댓글 필터링
- 단순 규칙으로 잡아내기 힘든 욕설, 혐오, 비방, 선정성 문구를 OpenAI API를 통해 판별
- run_llm 함수를 호출하여 맥락적 부적절성을 파악하고, 해당 인덱스의 데이터를 안전하게 제거
3) 전처리 결과 시각화
- 사용자가 전처리 과정을 추적할 수 있도록 단계별 로그(정적 전처리 전/후 개수, LLM 필터링 후 남은 개수)를 화면에 출력

**6. [F106] 텍스트 데이터 증강**
- 전처리가 완료된 깨끗한 댓글을 기반으로, 원본의 의미는 유지하면서 표현만 다른 확장 데이터를 생성
- ast.literal_eval을 사용하여 LLM의 응답을 파이썬 리스트 구조로 안전하게 파싱하여 처리

**7. [F107] 통합된 데이터 결과 출력**
- 정적/LLM 전처리를 거친 '정제 데이터'와 AI가 새롭게 생성한 '증강 데이터'를 하나로 통합하여 최종 데이터셋을 구축함
- 통합된 결과를 테이블(Table) 형태로 화면에 출력하여 사용자가 한눈에 원본과 증강본의 품질을 비교·확인할 수 있도록 시각화함

### 3. 기술 스택 및 라이브러리
- Framework: Django  
- Language: Python
- AI/API: OpenAI API (gpt-5-nano)  
- Environment: python-dotenv (API Key 및 모델 환경 변수 관리)  
- Frontend: Bootstrap 5

### 4. 학습 기록 및 회고
- 함수 설계 및 단일 책임 원칙(SRP)
    - preprocess_comments와 filter_inappropriate를 분리하여 각 함수의 역할을 명확히 하되, 상위 함수가 이를 조율(Orchestration)하게 하여 코드의 재사용성과 가독성을 높임

    - 함수 하나가 하나의 기능만 수행해야 한다는 원칙을 지키면서도, 전체 프로세스를 관리하는 구조적 설계를 고민함

- 데이터 무결성 및 효율적 처리
    - 리스트 요소를 삭제할 때 인덱스 밀림 현상을 방지하기 위해 reversed()를 사용하여 뒤에서부터 pop()하는 안정적인 삭제 로직을 구현함
    - LLM 호출 비용과 시간을 절약하기 위해, 비용이 저렴한 정적 전처리를 우선 수행하여 API로 전달되는 데이터 양을 최소화함

- 사용자 인터페이스(UI/UX) 및 로깅
    - 백엔드에서 일어나는 복잡한 전처리 과정을 사용자에게 투명하게 공개하기 위해, before_count와 after_count를 비교하여 데이터 정제율을 시각적으로 보여주는 출력 형식을 설계함
    - 이미지 예시를 바탕으로 Django 템플릿 시스템을 활용해 깔끔한 입력 폼과 결과 확인 화면을 구성함

- 문제 해결 및 트러블슈팅
    - LLM의 응답이 리스트 형식이 아닐 경우를 대비해 try-except 문과 타입 체크(isinstance)를 활용하여 시스템이 중단되지 않도록 방어적으로 코딩함
    - json.loads와 ast.literal_eval의 차이를 이해하고, 파이썬 리스트 형태의 텍스트 응답을 처리하는 데 적합한 방식을 선택함
