# 🚀 07-PJT: 프로젝트 기획 및 기술 설계 - TMI(Tailored Mind Insight)

## 📝 프로젝트 개요
- **프로젝트명:** TMI(Tailored Mind Insight)
- **진행 기간:** 2026.05.08
- **목표:** AI 사물 인식(Vision)과 생성형 AI(LLM)를 결합하여, 사용자의 일상 사물을 관심 테마(역사, 과학 등)와 연결해 주는 지식 큐레이션 서비스 기획 및 설계
- **주요 기술 스택 (예정):** Python, Django, PostgreSQL, PyTorch, Gemini API, Wikipedia REST API

---

## 💡 1. 단계별 학습한 내용

### ① 요구사항 정의 및 구체화 (문제 정의)
단순히 '재미있는 앱을 만들자'는 아이디어를 넘어, **SMART 원칙**을 적용하여 구체적인 목표로 변환하는 과정을 학습했습니다. 특히 기능적 요구사항(테마 설정, 사물 인식 등)과 비기능적 요구사항(API 호출 5초 이내 응답, 트래픽 최적화)을 분리하여 명세서를 작성함으로써 개발의 우선순위를 명확히 할 수 있었습니다.

### ② 데이터 수급 및 캐싱 전략 (데이터 탐색)
방대한 지식 데이터를 다루기 위해 **Wikipedia REST API**와 **Gemini API**의 무료 티어 활용 방안을 분석했습니다. 매 요청마다 API를 호출하면 발생하는 지연 시간과 비용 문제를 해결하기 위해, 한 번 생성된 TMI 데이터를 자체 DB(PostgreSQL)에 저장하고 재사용하는 **'스마트 캐싱(Smart Caching)' 전략**을 설계했습니다.

### ③ 시스템 구조 시각화 (다이어그램 설계)
기획한 아이디어를 개발 언어로 번역하기 위해 다이어그램을 작성했습니다.
- **ERD:** 유저의 관심도(`INTEREST`)와 지식(`KNOWLEDGE`), 그리고 좋아요(`LIKE`) 기능을 유기적으로 연결하는 관계형 데이터베이스 스키마를 구성했습니다.
- **Use Case Diagram:** 사용자의 사진 촬영 액션이 내부 DB 조회(`<<include>>`)와 외부 API 호출(`<<extend>>`)로 분기되는 로직을 시각화했습니다.

## 🚧 2. 어려웠던 부분 및 해결 과정H

### 외부 데이터 수집의 윤리 및 정책 파악
처음에는 위키피디아 데이터를 크롤링으로 무단 수집하려 했으나, `robots.txt` 정책과 서버 과부하 문제를 분석하게 되었습니다. 
결과적으로 크롤링 방식(`Disallow` 규정) 대신, 개발자를 위해 열려 있는 **공식 REST API**를 사용하고 HTTP Header에 `User-Agent`를 명시하는 방식으로 선회하여, 합법적이고 윤리적인 데이터 수급 파이프라인을 구축할 수 있었습니다.

### 복잡한 사용자 여정(User Flow)의 시각화
유저가 단순히 정보를 소비하는 것을 넘어, 테마를 선택하고 결과물을 도감에 저장하는 일련의 과정을 UI로 어떻게 풀어낼지 막막했습니다. 이를 해결하기 위해 기능 명세서에 작성된 흐름을 바탕으로 로그인부터 마이 도감까지 5단계의 화면 구조를 정의하고, AI 프롬프트를 활용해 와이어프레임의 초안을 구상하며 UI/UX 흐름을 확립했습니다.

## 🌟 3. 새로 배운 것들

- **Mermaid.js 활용:** ERD와 Use Case Diagram을 GUI 툴로 일일이 그리는 대신, 코드를 통해 다이어그램을 렌더링하고 관리하는 방법을 새롭게 익혔습니다. 이를 통해 문서의 버전 관리와 수정이 훨씬 용이해졌습니다.
- **Prompt Engineering for UI/UX:** 시각적 결과물을 얻기 위해 개발적인 설명보다는 'Glassmorphism', 'dribbble trending style'과 같은 디자인 특화 영문 키워드를 조합하여 프롬프트를 작성하는 방법을 배웠습니다.

## 🎯 4. 느낀 점

평소 프로젝트를 진행할 때 원인 분석과 완벽한 결과물 도출에 집착하는 경향이 있어, 개발 시작 전 기획 단계에서 완벽을 기하려다 시간을 지체하는 경우가 있었습니다. 하지만 이번 7회차 프로젝트를 통해 **'구조적 명확성 확보'**가 실제 코딩 단계에서의 낭비를 얼마나 줄여주는지 체감했습니다.

단순히 재미있는 아이디어(국장식 억지 논리 등)에서 출발했지만, 이를 실제 구현 가능한 데이터 스키마(ERD)로 쪼개고 API 연동 가능성(타당성 검토)을 분석해 나가는 과정이 매우 흥미로웠습니다. 특히 Python 백엔드(Django) 기술과 AI 모델(PyTorch, LLM)을 융합하는 아키텍처를 직접 설계해 보며, 앞으로 이어질 실제 구현 단계에 대한 강한 자신감과 동기부여를 얻을 수 있었습니다. 끝까지 파고드는 분석력을 바탕으로, 기획한 아키텍처를 실제 동작하는 코드로 오차 없이 구현해 낼 계획입니다.
